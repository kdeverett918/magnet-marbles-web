Last updated: 2026-06-22

## Target

Phone-first browser 3D arcade arena. Comparable quality bar:

- Marble It Up / toy-marble clarity: glossy readable marbles, clean table silhouette, instant motion readability.
- My Street marble minigame: collect, steal, bank, and goal ownership must be clear in the first round.
- Mobile arcade web games: first playable screen in a few seconds, low-chrome HUD, stable portrait controls, no confusing dead-end online flows.

AA web pass condition: a new mobile player can start from the menu, understand collect-to-bank within 30 seconds, finish a round without UI overlap, and return/rematch cleanly; desktop and mobile builds pass lint/build and screenshot smoke checks.

## Open Gaps

1. Public Render deployment is current by no-browser proof. `npm run live:version` and `npm run deploy:monitor` pass against matching live frontend/backend metadata, and the old `/audio/music.mp3` URL now serves only a `1,370` byte silent tombstone. This should stay monitored after any future deploy.
2. Colyseus 0.16 protocol compatibility remains a maintenance risk. Server audit is clean through `server/vendor/nanoid-compat`, but future agents should keep the shim covered by online smokes and migrate to an official matching 0.17+ server/client protocol once available.
3. Simulation and smoke coverage are stronger but still not complete for launch. The no-browser all-mode endurance soak catches finite-state and ownership regressions, and post-deploy disconnect takeover is proven by `npm run deploy:monitor`, but physical-device soak beyond deterministic sim and 15-second headless browser soak is still needed.
4. Mobile menu composition now passes the 390x844 screenshot smoke, but it still needs physical-phone review for glare, OLED contrast, and notch/safe-area variance.
5. Performance evidence is better but still incomplete. Normal, throttled mobile, and high-quality stress headless smokes pass, but launch still needs browser Performance panel or SpectorJS capture on midrange Android-class hardware to verify real GPU/postprocessing behavior and thermal stability.
6. Accessibility has repeatable keyboard smoke coverage for menu, gameplay, results, online retry states, Color Assist non-color identity markers, player Motion control, HUD objective announcements, and carried-marble meter semantics, but launch still needs a human screen-reader pass and physical-device checks for focus visibility under mobile browser UI.
7. Browser automation was stopped after Chrome tabs were reported as intrusive. The no-browser validation path is green, Chrome/CDP launch or tab creation now requires explicit `MM_ALLOW_BROWSER=1`, and `release:verify` performs static live preflight before any Chrome/CDP work, but the full browser gate still needs to be rerun deliberately before final release signoff.
