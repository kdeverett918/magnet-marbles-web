# Magnet Marbles Web Launch Readiness

Last updated: 2026-06-22

## Target

Phone-first browser 3D arcade arena. Comparable quality bar:

- Marble It Up / toy-marble clarity: glossy readable marbles, clean table silhouette, instant motion readability.
- My Street marble minigame: collect, steal, bank, and goal ownership must be clear in the first round.
- Mobile arcade web games: first playable screen in a few seconds, low-chrome HUD, stable portrait controls, no confusing dead-end online flows.

AA web pass condition: a new mobile player can start from the menu, understand collect-to-bank within 30 seconds, finish a round without UI overlap, and return/rematch cleanly; desktop and mobile builds pass lint/build and screenshot smoke checks.

## Open Gaps

1. Public Render deployment is not current. The local production preview and local authoritative server pass, but `https://magnet-marbles.onrender.com/` is still serving an older frontend that fails live verification, and `wss://magnet-marbles-server.onrender.com` is still serving an older/unversioned backend. Commit and deploy both local launch candidates, then rerun `npm run release:verify` as the public-release proof gate.
2. Online hosting is still a launch risk on Render free tier. The client now warms `/health`, explains/retries failures, and `npm run online:smoke` can verify live joins, but launch should still use an always-on backend plus scheduled or deploy-hook smoke monitoring.
3. Colyseus 0.16 protocol compatibility remains a maintenance risk. Server audit is clean through `server/vendor/nanoid-compat`, but future agents should keep the shim covered by online smokes and migrate to an official matching 0.17+ server/client protocol once available.
4. Simulation and smoke coverage are stronger but still not complete for launch. Post-deploy disconnect takeover is now wired into `release:verify`, but it cannot pass until the live backend is current; longer physical-device soak runs beyond the deterministic sim-level mode soak and 15-second headless browser soak are still needed.
5. Mobile menu composition now passes the 390x844 screenshot smoke, but it still needs physical-phone review for glare, OLED contrast, and notch/safe-area variance.
6. Performance evidence is better but still incomplete. Normal, throttled mobile, and high-quality stress headless smokes pass, but launch still needs browser Performance panel or SpectorJS capture on midrange Android-class hardware to verify real GPU/postprocessing behavior and thermal stability.
7. Accessibility has repeatable keyboard smoke coverage for menu, gameplay, results, and online retry states, but launch still needs a human screen-reader pass and physical-device checks for focus visibility under mobile browser UI.
8. Browser automation was stopped after Chrome tabs were reported as intrusive. The a11y smoke now has runtime-evaluation timeouts and no test/dev ports are left listening, but the full browser gate still needs to be rerun deliberately before deployment.
